**Key reference:** sfincs.inp physics/numerics keyword table (parameters.html, SFINCS 2.3.0 latest). Authoritative values verbatim from the manual: advection (default 1; ONLY values 0=SFINCS-LIE local-inertial / 1=SFINCS-SSWE advection-on; there is NO value 2); alpha (default 0.5, CFL dt-reduction, range 0.1-0.75); theta (default 1.0 = no smoothing, range 0.8-1.0); huthresh (default 0.05 m, min flow-depth wet/dry limiter); coriolis (default True — REAL keyword — but inert unless latitude!=0.0 on a projected CRS, i.e. latitude is the actual lever, default latitude=0.0); baro (default 1 = atmospheric-pressure/inverse-barometer term in momentum eqn); viscosity (default 1 = on) + nuvisc (default 0.01 per-m of cell length); wind drag curve cdnrb (default 3) / cdwnd [0 28 50] m/s breakpoints / cdval [0.001 0.0025 0.0015] coefficients (Vatvani 2012, as in Delft3D); qinf (default 0 mm/hr constant uniform infiltration) + qinf_zmin; scsfile (SCS Curve-Number max soil retention, no-recovery). Forcing: waterlevel boundary = bnd (point x,y) + bzs (slowly-varying WL time series) at msk=2 cells, with bzi as the SEPARATE quickly-varying short/IG-wave component (signal around 0, incoming-only); discharge = src (point x,y, may be interior) + dis (time series); precip = precipfile (uniform) / amprfile / netamprfile.

**Best practices:**
- Coriolis: expose the AOI-centre latitude (physics_registry coriolis_latitude->sfincs.inp:latitude) and SET it for any large-domain surge run — the manual is explicit that coriolis is on-by-default but does nothing while latitude=0.0 on a projected CRS, so a surge deck that omits latitude silently runs WITHOUT Coriolis. Default the composer to the bbox-centre latitude when wind/surge forcing is present.
- Tsunami: force the long-period wave through bzs (slowly-varying), never bzi — bzi is reserved for short/infragravity waves whose signal is centred on 0 with the reflected component removed internally. The msk=2 boundary already supplies the weakly-reflective Riemann condition, so no extra absorbing-BC config is needed. For long-wave fidelity, allow theta<0.8 (Tohoku used 0.54) and a sub-minute output cadence; the design-doc:137 already flags the 60s output floor as a tsunami under-sampling risk — keep that.
- Infiltration precedence: setup_cn_infiltration pops the default qinf and writes scsfile, so CN and constant-qinf are mutually exclusive — enforce CN-wins precedence (design-doc:63,134 already does) and never emit both. For a single-band GCN250 CN raster pass antecedent_moisture=None (the bare-DataArray branch); the default 'avg' expects a cn_avg variable inside a Dataset and will ValueError on a bare band (design-doc:12,133 — correct).
- Keep advection ON (1) as the default for SSWE momentum (manual: 'Recommended is to turn the advection term always on'); only drop to 0 (SFINCS-LIE) deliberately for cheap/large local-inertial runs. Never emit a value other than 0 or 1.
- alpha is the CFL safety lever (0.1-0.75, default 0.5); huthresh wet/dry limiter (0.001-0.1, default 0.05). Keep registry ranges inside the manual's recommended bands and label the 'default' fields with the manual's actual engine defaults so applied_physics_delta narration ('from X to Y') reports the true engine baseline.
- baro=1 (default) already activates the atmospheric-pressure/inverse-barometer term, so a pressure-forcing run needs NO new key — the design-doc:34 correctly notes this. Don't add a redundant 'baro' physics-registry entry; pressure forcing (spwfile/netampr) is what supplies the field, baro just enables the term.


## GeoClaw (Clawpack) — tsunami inundation / shallow-water solver (confidence high)

**Docs read:** https://www.clawpack.org/geoclaw.html (GeoClaw overview + component link map); https://www.clawpack.org/setrun_geoclaw.html (setrun.py parameters: geo_data, topo_data, dtopo_data, qinit_data, refinement_data, regiondata.regions, fgmax_data, surge_data); https://www.clawpack.org/dtopotools_module.html (SubFault/Fault classes, create_dtopography, okada, coordinate_specification, units); https://www.clawpack.org/okada.html (Okada source: strike/dip/rake/slip/length/width/depth, coordinate_specification conventions, Green's-function half-space model, create_dtopo_xy workflow); https://www.clawpack.org/dtopo.html (dtopotype 1/3 file formats, dt_max_dtopo); https://www.clawpack.org/topotools_module.html (Topography class, topotype 1/2/3, nested DEM stitching, ESRI-ASCII header); https://www.clawpack.org/fgmax.html (FGmaxGrid attributes, num_fgmax_val 1/2/5, arrival_tol, point_style, fgmax output files, read_output); https://www.clawpack.org/gauges.html (gaugedata.gauges append [gaugeno,x,y,t1,t2], gaugeNNNNN.txt output); https://www.clawpack.org/quick_tsunami.html (canonical tsunami workflow: chile2010 example, maketopo.py/setrun.py/setplot.py/Makefile, dtopo+nested-topo+AMR-regions+fgmax+gauges); https://depts.washington.edu/clawpack/links/noaa-tsunami-benchmarks/ (NTHMP/NOAA benchmark suite GeoClaw validates against: Monai Valley, Okushiri, conical island, etc.)


**Key reference:** Canonical GeoClaw tsunami-inundation setrun.py has FIVE load-bearing data blocks, and the inundation answer (max depth + arrival time) comes from fgmax, NOT from the fort.q frames. Authoritative key/semantics map:

geo_data (rundata.geo_data): gravity=9.81; coordinate_system=2 (lat/lon on the sphere) [1=Cartesian meters]; earth_radius=6367.5e3; sea_level (relative to the topo's 0 datum); dry_tolerance (~1e-3); friction_forcing=True; manning_coefficient (scalar OR list with manning_break); friction_depth (only apply friction where h<friction_depth).

topo_data.topofiles: append [topotype, fname]; topotype 3 = ESRI-ASCII grid (header ncols/nrows/xll{corner|center}/cellsize/nodata_value). NESTED topo is the norm: multiple topofiles at increasing resolution; GeoClaw prioritizes the FINEST DEM where they overlap (coarse ocean bathymetry + fine coastal DEM).

dtopo_data.dtopofiles: append [dtopotype, fname] (dtopotype 3 recommended; header mx,my,mt / xlower,ylower,t0 / dx,dy,dt then mt blocks of my lines x mx dz values). dt_max_dtopo caps the timestep WHILE the seafloor is moving.

dtopotools (the Okada source): SubFault attributes = strike (deg CW from N, 0-360), dip (0-90), rake (deg CCW from strike, -180..180), slip (m), length (m), width (m), depth (m below seafloor of the reference point), longitude, latitude, mu (rigidity, Pa), coordinate_specification in {'top center','bottom center','centroid','noaa sift','top upstrike corner'}. Workflow: build SubFault(s) -> Fault(); fault.subfaults=[...]; x,y = fault.create_dtopo_xy(dx=1/60., buffer_size=2.0); fault.create_dtopography(x,y,times=[...]); fault.dtopo.write('f.tt3', dtopo_type=3). Okada is a Green's-function elastic-half-space model (Poisson ratio 0.25). Mo = mu*slip*(length*width); Mw=(2/3)log10(Mo)-6.07 (fault.Mo()/fault.Mw() compute these).

amrdata + regiondata.regions: amr_levels_max + refinement_ratios_x/y/t (per-level lists, NOT all 2s); flag2refine + refinement_data.wave_tolerance (flag cell if |eta-sea_level|>tol) + speed_tolerance (list, per-level speed flag). regiondata.regions.append([minlevel,maxlevel,t1,t2,x1,x2,y1,y2]) FORCES fine AMR over the coastal inundation zone for the whole run — this is how the canonical workflow gets resolution at the coast.

fgmax_data.fgmax_grids (THE inundation monitor): FGmaxGrid with point_style (2 = rectangular grid: x1,x2,y1,y2,dx,dy or nx,ny), tstart_max, tend_max, dt_check, min_level_check, arrival_tol (wave 'arrived' when eta>sea_level+arrival_tol), interp_method (0 cell-average recommended). num_fgmax_val: 1=max depth h; 2=h+speed; 5=h+speed+momentum+momentum-flux+min-depth. Output fgmaxNNNN.txt has lon,lat,AMR-level,topo B, max h (and speed/flux), the TIME of each max, and the ARRIVAL TIME (final column). Read with fgmax_tools.FGmaxGrid().read_output(). Inundation depth on land = max h where B>sea_level.

gaugedata.gauges.append([gaugeno,x,y,t1,t2]) -> gaugeNNNNN.txt time series of q at a point (validation/observed-vs-computed).

**Best practices:**
