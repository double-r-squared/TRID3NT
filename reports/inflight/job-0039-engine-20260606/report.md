## Decisions Made

- **Decision: NLCD access tier is Tier 2 (OGC service — MRLC WMS) NOT the kickoff-inferred Tier 3.** Rationale: live verification 2026-06-07 revealed (a) MRLC's direct file mirror at `s3-us-west-2.amazonaws.com/mrlc/Annual_NLCD_LndCov_<YEAR>_CU_C1V0.tif` is a 42-byte placeholder stub for the 2023 file URL (the byte content is a 1×1 TIFF IFD with two `0xFFFFFFFF` strip offsets — NOT a real raster), and the 2019/2021 file URLs at the same path return HTTP 403; (b) the MRLC WCS endpoint at `/geoserver/mrlc_display/wcs` accepts a TCP connection but times out before serving GetCapabilities; (c) the MRLC WMS GeoServer at `www.mrlc.gov/geoserver/mrlc_display/wms` lists NLCD year layers `NLCD_<YEAR>_Land_Cover_L48` and supports `GetMap?format=image/geotiff`, returning real raster bytes (tested 67,520-byte GeoTIFF for a Fort Myers bbox — verified header is `MM\x00*` little-endian TIFF magic). Live-verified tier wins per §F.1.1 discipline; the docstring records Tier 2 and the OQ surfaces the deviation. Alternatives considered: keep the kickoff's Tier 3 fiction and call the 42-byte stub (rejected — that's the silent-wrong-answer pattern Invariant 7 bans); fall back to MRLC's `mrlc_zip_metadata_api` for download URLs (rejected — those return zip files, not GeoTIFFs; same Tier 4 download-then-clip cost as NHDPlus HR but for a 4-GB CONUS file, much worse than per-bbox WMS GetMap). Surfaced as **OQ-39-NLCD-TIER-DEVIATION**.

- **Decision: NLCD vintage year default is 2021, not 2023.** Rationale: live MRLC GetCapabilities probe 2026-06-07 lists discrete-year NLCD CONUS L48 WMS layers up through `NLCD_2021_Land_Cover_L48`; the 2023 release (Annual NLCD Collection 1.0) is in the catalog under change-detection summary layers (`Annual_NLCD_Smy_LndCov_ChgCnt_1985_2023_CU_C1V0` etc.) but no discrete `NLCD_2023_Land_Cover_L48` layer is present. The substrate defaults to 2021 because (a) it's the newest discrete-year layer with a stable WMS endpoint, (b) it aligns with the OQ-4 §4 Manning's-mapping CSV vintage that job-0042 will author, (c) the dataset string parameter (`"nlcd_2021"`, `"nlcd_2019"`, …) makes opting into older vintages trivial when needed. When MRLC publishes `NLCD_2023_Land_Cover_L48`, a one-line table addition in `_NLCD_WMS_LAYER_BY_YEAR` plus a default-flip in the signature lands the bump. Surfaced as **OQ-39-NLCD-VINTAGE-DEFAULT**.

- **Decision: `fetch_landcover` returns a `dict[str, Any]` (layer + vintage sidecar) NOT a bare `LayerURI`.** Rationale: the kickoff's example syntax `LayerURI.metadata["nlcd_vintage_year"] = 2021` was illustrative — `LayerURI` is in `packages/contracts/grace2_contracts/execution.py` (FROZEN per kickoff §file ownership) with `model_config = ConfigDict(extra="forbid", ...)` from `GraceModel`, so `metadata` is NOT a valid field and cannot be added without amending the contract. The pragmatic alternative is a dict return shape: `{"layer": LayerURI, "nlcd_vintage_year": 2021, "dataset": "nlcd_2021", "source": "mrlc-wms"}`. This mirrors the existing `geocode_location` and `lookup_precip_return_period` patterns (also dict returns without a contract). Downstream consumer (job-0042 `build_sfincs_model`) reads `result["layer"]` for the `LayerURI` and `result["nlcd_vintage_year"]` for the validation gate. Surfaced as **OQ-39-LANDCOVER-RETURN-SHAPE-CONTRACT-PROMOTION** for schema to consider promoting (a `LandcoverFetchResult` pydantic model with `layer: LayerURI` + `nlcd_vintage_year: int | None` + provenance dict).

- **Decision: ESA WorldCover branch is a placeholder; NLCD is the v0.1 default + only-implemented landcover source.** Rationale: the kickoff calls for ESA WorldCover opt-in via `dataset="esa_worldcover_2021"`; the implementation reserves the dispatch path and the shape (`_fetch_esa_worldcover_bytes` raises `UpstreamAPIError` until the Planetary Computer STAC branch lands). Live verification confirmed ESA WorldCover IS in the Planetary Computer STAC catalog (`esa-worldcover` collection — observed in the 2026-06-07 probe alongside `nrcan-landcover` Canada Land Cover). The full implementation is a follow-up because the Manning's mapping CSV for ESA WorldCover classes differs structurally from NLCD's (different class enumeration, e.g. "Tree cover", "Grassland", "Built-up" vs NLCD's "Deciduous Forest", "Pasture/Hay", "Developed, Medium Intensity"); job-0042 owns the Manning's mapping for NLCD first, and the ESA branch must wait until either the M5 demo demands non-CONUS coverage (Fort Myers is CONUS) or HydroMT requires it. Surfaced as **OQ-39-ESA-WORLDCOVER-SUBSTRATE**.

- **Decision: NHDPlus HR access tier is Tier 4 (region download + local clip) — matches the kickoff inference.** Rationale: live verification 2026-06-07 confirmed the kickoff's inference. The TNM staged-products S3 bucket at `prd-tnm.s3.amazonaws.com/StagedProducts/Hydrography/NHDPlusHR/Beta/GDB/NHDPLUS_H_<HUC4>_HU4_GDB.zip` serves HUC4-scoped FileGDB zip files (HUC4 0309 South Florida is 151,111,923 bytes — 144 MB), HTTP 200 with `accept-ranges: bytes`. No per-bbox query API exists for NHDPlus HR raw geometry. The implementation downloads the HUC4 GDB on every cache miss, extracts via `zipfile`, reads `NHDFlowline` via `geopandas.read_file(layer="NHDFlowline", bbox=bbox)`, clips with `shapely.geometry.box`, writes a FlatGeobuf, and returns the bytes through `read_through`. The **two-stage cache** the kickoff suggests (cache the HUC4 region file separately at `cache/static-30d/river_geometry/_regions/<huc4>.gdb.zip`, share across all per-bbox clips inside that region) is OQ-39-NHDPLUSHR-TWO-STAGE-CACHE — the simpler single-stage cache covers the M5 Fort Myers demo path (one cache miss = one ~144 MB download + ~30-day cache window per quantized bbox); the two-stage optimization is a follow-up worth ~144 MB of saved bandwidth per cold-bbox call within the same HUC4. Surfaced as **OQ-39-NHDPLUSHR-TWO-STAGE-CACHE**.

- **Decision: HUC4 routing uses a heuristic envelope table (8 entries), not a real WBD point-in-polygon.** Rationale: same trade-off as the job-0033 `_state_fips_for_lonlat` and job-0037 `_iso3_for_lonlat` heuristics — a real point-in-polygon needs a cached WBD HU4 shapefile + `shapely`/`geopandas` lookup, fine for production but more than this job needs (the M5 demo target is Fort Myers, FL — HUC4 0309). The 8-entry table covers the M5 demo plus several sprint-08 / sprint-09 candidate scenarios (Hurricane Harvey Houston / Katrina Louisiana / Sandy NYC / Florence Cape Fear / wildfires LA basin), enough for any planned smoke runs. Replacement with WBD admin0 is a tracked follow-up alongside the other two heuristic follow-ups. Surfaced as **OQ-39-NHDPLUSHR-HUC4-ROUTING-HEURISTIC**.

- **Decision: Atlas 14 access tier is Tier 3 (direct HTTPS point query, CSV response) — matches the kickoff inference.** Rationale: live verification 2026-06-07 confirmed the kickoff's inference. `hdsc.nws.noaa.gov/cgi-bin/hdsc/new/fe_text_mean.csv?lat=&lon=&data=depth&units=english&series=pds` returns HTTP 200 with a 1.5-KB CSV containing the duration × ARI matrix of precipitation depths plus a `NOAA Atlas 14 Volume <N> Version <V>` provenance line. Body content-type is `text/html` despite the URL suffix `.csv`, but the body shape is CSV-like (we parse it via a custom parser that handles the header section + matrix rows). Fort Myers (lat=26.6, lon=-81.9) resolved to "NOAA Atlas 14 Volume 9 Version 2" / "Project area: Southeastern States" with 100-yr 24-hr depth = 12.1 inches.

- **Decision: Atlas 14 single-point lookup, NOT a partial-bbox fetch.** Rationale: Atlas 14 PFDS is a per-coordinate query endpoint — no native bbox lookup exists. For a small bbox (Fort Myers `(-81.92, 26.55, -81.80, 26.68)` ≈ 200 km²), the precipitation frequency is essentially uniform across the bbox (Atlas 14 isohyets are smooth at this scale); a single point lookup at the bbox center is sufficient for SFINCS pluvial forcing reconstruction. For larger bboxes the per-point cost adds up; a future tiled-lookup workflow could fan out N×M calls at the Atlas 14 grid intersection points and aggregate. The substrate exposes a single-point parameter surface (`location=(lat, lon)`) rather than a `bbox=(...)` — matching the natural Atlas 14 grain. Surfaced as **OQ-39-ATLAS14-SINGLE-POINT-VS-BBOX** (TENTATIVE: single-point default, fan-out workflow on demand).

- **Decision: bbox quantization per-source-meaningful (NLCD 30 m, NHDPlus HR HUC4-scoped + 10 m bbox snap, Atlas 14 1/120-degree).** Per the OQ-32-QUANTIZATION-LOCATION engine-side resolution. Each tool snaps its bbox / location to its native grid before handing `params` to `read_through`. NLCD: `round_bbox_to_resolution(bbox, 30)` (native 30 m). NHDPlus HR: `round_bbox_to_resolution(bbox, 10)` for cache-edge stability + cache key explicitly includes the HUC4 code so two disjoint HUC4s never collide. Atlas 14: `_quantize_lonlat_to_atlas14_grid(lat, lon)` snapping to the nearest 1/120-degree grid intersection.

- **Decision: NLCD GeoTIFF byte fetch via WMS GetMap (Tier-2 OGC, byte-materialized) writes through the FR-DC-3 cache as if it were a Tier-1 COG.** Rationale: §F.1.1 Tier 2 normally means "layer reference IS the URL — QGIS Server proxies / re-renders, no GCS bytes cached." But for SFINCS setup (job-0042 `build_sfincs_model`) HydroMT's roughness component needs raster bytes, not a WMS reference. The fetcher pulls a GetMap GeoTIFF for the bbox at 30 m, writes it to the FR-DC-3 cache as a GeoTIFF artifact. The cache shim doesn't care about the access tier on its input; the on-disk artifact is a real GeoTIFF either way. This is documented in the tool docstring and called out as a structural departure from the "Tier 2 bypasses the cache" rule of thumb. The Discovery-First lane pattern (FR-PHC) still applies as `fetch_public_hazard_layer(catalog_entry_id, bbox?)` does NOT cache layer bytes — that's a different tool with a different role.
