grace2 conda env (QGIS 3.40.3-Bratislava) — a single-layer EPSG:4326 CONUS
project carrying an OSM XYZ basemap named `basemap-osm-conus`; a matching QML

routing to a follow-up infra job. Engine deliverables (1) — (5) are complete;

## Decisions Made

- Decision: **PyQGIS scripting (`build_sample_project.py`) instead of QGIS
  Desktop GUI for authoring.**
  - Rationale: reproducibility. No GUI session state, no manual click trail;
    the script is the provenance, the `.qgs` is the snapshot. Re-running
    regenerates a bit-stable file. Mirrors the engine discipline
    "PyQGIS-worker is the only `.qgs` writer" — this job authors the initial
    `.qgs` once via a deterministic PyQGIS script and uploads it as the
    starting state; all subsequent mutations (job-0020) go through the worker
    round-trip.
  - Alternatives considered: QGIS Desktop in the grace2 env (kickoff's
    TENTATIVE). Rejected because the grace2 env is headless on this Debian
    box and a GUI session would not survive across machines.
- Decision: **OSM XYZ basemap layer (kept), NOT a FlatGeobuf fallback.**
  - Rationale: the kickoff's TENTATIVE is OSM XYZ; the FlatGeobuf alternative
    would only mitigate a per-layer render failure (XYZ tiles unreachable
    from the QGIS Server's egress). The live failure surfaced in this job is
    **NOT a layer-render failure** — it is a **project-file-load failure
    (QGIS Server cannot open the `.qgs` from `/vsigs/` at all)**, which is
    layer-agnostic. Switching the layer source would not change the project
    loader's behavior. The TENTATIVE OSM XYZ choice stands; the infra gap
    is surfaced as the dominant Open Question (OQ-19A).
  - Alternatives considered: static CONUS-polygon FlatGeobuf committed
    alongside (per kickoff alternative). Rejected per the diagnosis above —
    it would not unblock the live verification.
- Decision: **Layer name `basemap-osm-conus`.**
  - Rationale: matches the directive in the live engine kickoff text from
    the orchestrator ("Layer name `basemap-osm-conus` so the QML preset matches
    by name"). This differs from `osm-basemap` used in the
    `reports/inflight/job-0019-engine-20260605/audit.md` Acceptance Criteria
    section, which references `<Name>osm-basemap</Name>`. Surfaced as OQ-19B.
    The QML file's `<comment>` block names `basemap-osm-conus` to match.
  - Alternatives considered: use the audit.md name `osm-basemap`. Rejected
    in favor of the live orchestrator directive, which is the more specific
    and recent instruction.
- Decision: **QML preset is a raster `multibandcolor` renderer with opacity +
  brightness + hue/saturation tweaks** (rather than a single-band pseudocolour
  ramp or a vector polygon symbol).
  - Rationale: the underlying `basemap-osm-conus` layer is a 3-band RGB XYZ
    tile source, so a single-band ramp is not meaningful. The renderer is
    the simplest one that has a visible effect on an RGB raster (slight
    background dimming so future overlays read against it).
  - Alternatives considered: empty pipeline (no renderer override). Rejected —
    a stub QML must actually exercise the apply-preset codepath, which means
    the renderer block must be non-trivial.
- Decision: **`styles/basemap.qml` content authored here, but NOT triggering
  a QGIS Server image rebuild + redeploy in this job.**
  - Rationale: the kickoff allows either trigger-here or defer-to-job-0021;
    the dominant Open Question OQ-19A (QGIS Server `/vsigs/` MAP gap) makes
    a rebuild-and-redeploy ineffective for the live verification anyway.
    Surfaced as OQ-19C; recommendation is to bundle the QML bake into the
    follow-up infra fix that addresses OQ-19A.
  - Alternatives considered: rebuild + redeploy now (kickoff TENTATIVE).
    Deferred to the same follow-up that closes OQ-19A.

## Open Questions

### OQ-19A (BLOCKING the live verification — infra follow-up required)

**QGIS Server cannot open the `.qgs` from `/vsigs/` in the live Cloud Run
service deployed by job-0018.** All live WMS requests (`GetCapabilities`,
`GetMap`) against the uploaded `gs://grace-2-hazard-prod-qgs/grace2-sample.qgs`
return HTTP 500 with the QGIS Server log line:

```
CRITICAL Server[18]: Error when loading project file '/vsigs/grace-2-hazard-prod-qgs/grace2-sample.qgs': Unable to open /vsigs/grace-2-hazard-prod-qgs/grace2-sample.qgs
```

Diagnosis (per the "Diagnose before fix" cross-cutting principle — failing
layer named):

- Bucket IAM verified — `grace-2-qgis-server@…` has
  `roles/storage.objectViewer` on `gs://grace-2-hazard-prod-qgs/` (job-0018
  binding). Not a permissions issue at the GCS layer.
- The `.qgs` file is intact in GCS — MD5 matches local; `gcloud storage ls`
  returns it; `gcloud storage cat` (not run here, but the upload succeeded)
  would read it back. Not a payload issue.
- The QGIS Server container has no environment variables related to GDAL VSI
  auth (`CPL_*`, `GS_*`, `GOOGLE_APPLICATION_CREDENTIALS`) per
  `gcloud run services describe`. GDAL on GCE/Cloud Run autoconfigures
  `/vsigs/` from the metadata server in GDAL ≥3.7, but two paths can break
  this:
  1. The `.qgs` project loader in QGIS Server's `QgsProject::read()` may use
     Qt's `QFile` (not GDAL VSI) for the project file itself — `QFile` does
     not understand the `/vsigs/` prefix and treats it as a local-filesystem
     path, which does not exist.
  2. Even if QGIS Server's project loader dispatches through GDAL VSI for
     VSI prefixes (a feature added in QGIS 3.20+), the GDAL build in the
     `qgis/qgis-server:3.40.15-noble` base may not be auto-detecting the
     Cloud Run metadata server, or may need an explicit
     `CPL_MACHINE_IS_GCE=YES` / `CPL_GS_USE_INSTANCE_PROFILE=YES` hint.

Candidate fixes (infra follow-up):

- **(a)** Add a pre-handler / sidecar that fetches the requested `.qgs` from
  GCS into a local `/tmp/qgs-cache/` directory before passing the local path
  to QGIS Server. Simplest, most portable, no upstream dependence.
- **(b)** Mount the `grace-2-hazard-prod-qgs` bucket via `gcsfuse` as a
  read-only FUSE filesystem at `/mnt/qgs/` and use `MAP=/mnt/qgs/grace2-sample.qgs`.
  Cloud Run supports `gcsfuse` mounts natively in second-gen execution
  environment.
- **(c)** Try setting GDAL VSI env vars on the Cloud Run service
  (`CPL_MACHINE_IS_GCE=YES`, `CPL_GS_USE_INSTANCE_PROFILE=YES`) and verify
  whether QGIS Server's project loader actually dispatches through GDAL VSI
  for `/vsigs/` MAP=. Lowest-cost diagnostic step.

TENTATIVE recommendation: try **(c)** as a 5-minute diagnostic to confirm
whether the issue is GDAL VSI auth or QGIS Server's project loader path;
if (c) fails, ship **(b)** — gcsfuse is the canonical pattern for serving
.qgs from GCS to QGIS Server on Cloud Run.

Routing: open a follow-up infra job (call it `job-NNNN-infra` post-0023);
gates the M2 exit criteria #1 + #3 + the M2 acceptance suite (job-0023)
from passing. SRS refs: FR-QS-1, FR-QS-2, FR-MP-3, Invariant 4.

### OQ-19D (NON-BLOCKING — OSM XYZ tile-usage-policy implication)

QGIS Server, when it eventually does load this `.qgs` (post-OQ-19A fix),
will egress tile requests to `https://tile.openstreetmap.org/...` from the
Cloud Run instance's outbound IP — the OSM tile-usage policy applies. At M2
smoke scale (single-tile load per `GetMap` request) this is fine; in M3 when
the web client consumes tiles, the volume profile changes. SRS FR-DT-1 is
satisfied by OSM as the v0.1 Tier A reference; the OSM-policy-vs-self-hosted
trade-off comes up later.

Recommendation: defer to M3 web-tile-consumption job; add a
`HTTP-User-Agent: GRACE-2-QGIS-Server/0.1` to the XYZ source URI if OSM ever
rate-limits.

### OQ-19E (NON-BLOCKING — `_attachments.zip` sidecar handling)

`QgsProject.write()` automatically emits a `<projectname>_attachments.zip`
sidecar (an empty styles.db wrapper in this case). The build script deletes
this from the on-disk path after write to avoid committing a useless sidecar.
QGIS Server does not require the sidecar to render. If a future preset / form
or embedded resource is added to the project, the sidecar will carry it and
must be committed alongside the `.qgs`.

Recommendation: keep the current "delete the empty sidecar" behavior implicit
(handled out-of-band by `rm`; absorb into the build script in a follow-up
minor edit if attention shifts here).
