## SFR river-aquifer exchange (MODFLOW 6)

MODFLOW 6 SFR (streamflow routing) upgrade for TRID3NT local: replaces/augments the existing RIV head-dependent river boundary (sprint-17 J9 river-seepage path in services/workers/modflow/gwt_adapter.py) with a routed stream network, unlocking the "how does pumping this well affect the river" story. SFR computes stage and discharge PER REACH from Manning flow (RIV only knows a fixed stage and an exchange flux), so the model can answer streamflow depletion (baseline-vs-pumped delta in reach flow, attributable per reach), gaining/losing classification that is hydraulically consistent, baseflow accounting, and "will the reach run dry" low-flow screening. It composes with the existing WEL archetype (sustainable_yield becomes stream_depletion when river-coupled) and later with the RCH archetypes (MAR/wetland baseflow-gain reporting; regional_water_budget picks up the SFR term from the cbc with minor name normalization in _read_cbc_budget_partition). Reach geometry reuses the proven NHDPlus seams: agent-side fetch_river_geometry (NHDPlus HR primary, OSM fallback) -> resolve_river_polyline_lonlat (run_modflow.py:508, longest merged LineString, <=200 vertices) -> _drape_polyline_onto_grid (gwt_adapter.py:541, returns (row,col,reach_len_m) in path order - exactly the ordered-reach input SFR needs); the TELEMAC worker's fetch_river_centerline (telemac_river_dye_build.py:280, NLDI COMID-snap + navigate + _stitch_flowlines) is the pattern for optional reach extension via the existing fetch_nhdplus_nldi_navigate tool - reuse the DATA path, do not import worker code across engines. All solves run through the existing local mf6 seam (GRACE2_MODFLOW_LOCAL=1, bin/mf6 6.5.0 which ships SFR6 - proven by smoke run), the existing convergence guard (mfsim.lst authoritative), and the existing solver-confirm/granularity gate hook noted at run_modflow_tool.py:23. ## GAIA sediment transport (TELEMAC)

GAIA is TELEMAC's sediment-transport/morphodynamics module, fully compiled in our trid3nt-local/telemac:latest image (libgaia4telemac2d.so + gaia.dico present; no image rebuild). It runs internally coupled to TELEMAC-2D via two keywords in the existing t2d_river.cas and its own small steering file. It supports non-cohesive (NCO, sand/gravel: bedload via 10+ formulas incl. Meyer-Peter-Muller/van Rijn/Wilcock-Crowe, plus suspension with Stokes/Zanke/van-Rijn auto settling) and cohesive (CO, mud: Krone deposition / Partheniades erosion, settling velocity per class) classes, a multilayer bed model, and writes its own SELAFIN with CUMUL BED EVOL (m), QS BEDLOAD (kg/m/s), and per-class concentrations. Critically, lecdon_telemac2d_gaia.f maps GAIA suspended classes onto T2D point-source tracers (TSCE=SEDSCE), so our proven finite-pulse point-source seam (SOURCES FILE with Q shutoff) injects a sediment slug exactly like the dye slug. RECOMMENDED v1 (simplest honest mode): one suspended sediment class, supply-limited bed (LAYERS INITIAL THICKNESS = 0 so nothing erodes; only the injected pulse can deposit), bed evolution output ON. This gives real settling + a genuine deposition map ('where and how much') without bed-erosion tuning; morphodynamic feedback is physically negligible at mm-scale evolutions. Pure-passive mode (relabeling the existing dye tracer) is rejected as dishonest for a deposition question - it has no settling. Full erodible-bed morphodynamics (initial thickness > 0 + bedload formulas) is the v2 flag, not v1.

**Engine config:** t2d_river.cas additions (2 lines, in author_deck's substance branch like the oil class): COUPLING WITH = 'GAIA'; GAIA STEERING FILE = gaia_river.cas. New worker-authored gaia_river.cas (~20 lines): GEOMETRY FILE = river.slf (same mesh); BOUNDARY CONDITIONS FILE = river.cli (reuse - GAIA's dico marks the BC file OBLIG and the CONLIM reader is the same format); RESULTS FILE = gaia_river.slf; VARIABLES FOR GRAPHIC PRINTOUTS = 'B,E,M,QSBL,CS1'; CLASSES TYPE OF SEDIMENT = 'NCO' (or 'CO' for a mud preset); CLASSES SEDIMENT DIAMETERS = d50_m (default 200e-6); CLASSES SEDIMENT DENSITY = 2650; CLASSES INITIAL FRACTION = 1.0; CLASSES SETTLING VELOCITIES = -9. ## WAQTEL water quality (TELEMAC)

WAQTEL is TELEMAC's water-quality kernel, ALREADY compiled into the telemac2d executable in trid3nt-local/telemac:latest (libwaqtel4telemac2d.so; opentelemac v9.0.0). It is activated from the telemac2d .cas with three keywords: COUPLING WITH = 'WAQTEL', WAQTEL STEERING FILE = <file>, WATER QUALITY PROCESS = <int>. WATER QUALITY PROCESS is a MULTIPLICATIVE PRODUCT OF PRIMES selecting modules (verified in telemac2d.dico + source_waq.F): 2=O2 (dissolved oxygen; 3 tracers O2/organic-load/NH4 + reaeration K2), 3=BIOMASS (5 tracers), 5=EUTRO (~8-tracer eutrophication set), 7=MICROPOL (micropollutant C + suspended/bed sediment + absorbed fractions, needs sediment coupling), 11=THERMIC (1 TEMPERATURE tracer + heat exchange with atmosphere), 17=first-order/T90 tracer DEGRADATION law, 19=ice-ghost. Combine by multiplying (22 = 2x11 runs O2+THERMIC together). Each module AUTO-ADDS its own tracers via nametrac_waqtel; degradation (17) is special - its nametrac branch does DO I=1,NTRAC and applies a decay sink to ALL EXISTING user tracers, so it rides directly on our current DYE tracer with ZERO new tracers. Recommended v1 as the 'third substance class' beside dye and oil: (a) first-order DECAY (WAQPROCESS=17) as the true simplest honest add - a ~3-line telemac2d .cas addition plus a ~4-line WAQTEL steering file layered on the UNCHANGED dye pipeline (no new tracer, no postprocess or contract change); (b) THERMIC (WAQPROCESS=11) as the flagship thermal-discharge demo - heavier (new TEMPERATURE tracer, constant ambient meteo, thermal colormap, degC). DO/O2, MICROPOL, BIOMASS, EUTRO are explicitly v2+ (they need BOD+reaeration, suspended-sediment transport, or the full nutrient set).
