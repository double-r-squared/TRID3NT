## Decisions Made

- **Decision: WorldPop REST country file (1km Aggregated product) is the WorldPop branch's data source, not a STAC.** Rationale: the kickoff suggested a STAC endpoint (Microsoft Planetary Computer's `worldpop-100m` collection or the WorldPop Hub STAC at `https://hub.worldpop.org/stac/`), but neither catalog exists at this writing — Microsoft Planetary Computer has 134 collections, none containing "worldpop"; the WorldPop Hub STAC URL returns HTTP 404. The de-facto canonical surface is WorldPop's REST index at `https://www.worldpop.org/rest/data/pop/<alias>?iso3=<ISO3>` which links to the country GeoTIFFs under `https://data.worldpop.org/`. Surfaced as **OQ-37-WORLDPOP-STAC-ENDPOINT**. Alternatives considered: pystac-client + Planetary Computer (rejected — collection does not exist); the WorldPop Hub STAC (rejected — 404); AWS Open Data S3 mirror (rejected — `wpdata.s3.amazonaws.com` returns 403, no public mirror configured).

- **Decision: 1km Aggregated product, not the 100m UN-adjusted product.** Rationale: WorldPop's HTTP server does NOT support range requests — `requests.get(url, headers={'Range': 'bytes=0-1023'})` returns HTTP 200 with the full body, not HTTP 206 Partial Content. GDAL's `/vsicurl/` driver consequently raises `Range downloading not supported by this server!` and a remote windowed read is impossible. With range requests unavailable, the choice becomes: download the entire country file per cache miss vs use a smaller product. The 100m UN-adjusted USA file is 4 GB; the 1km Aggregated USA file is 50 MB. The 30-day cache TTL means each cache miss costs one download, but a 4 GB download per miss is impractical for the agent runtime and acceptance loop. The 1km product is sufficient for M5 Fort-Myers-class bbox scales (exposure summation at bbox scale, not per-100m-cell analysis). Surfaced as **OQ-37-WORLDPOP-RESOLUTION-VS-RANGE**. Alternatives considered: (a) full 4 GB download per miss — rejected on tractability; (b) `/vsicurl/` windowed read of 100m — rejected because the server lacks range support; (c) Microsoft Planetary Computer windowed read — rejected because no WorldPop collection on PC.

- **Decision: vintage year 2020, not "latest available" autoflip.** Rationale: the WorldPop catalog has discrete vintages — `Global_2000_2020` (canonical analytical product, ends at 2020) and a newer `R2024B` / `R2025A` tree under different alias names with different file conventions. The `_2020` suffix is hardwired into the URL pattern `Global_2000_2020_1km/2020/USA/usa_ppp_2020_1km_Aggregated.tif`; future vintages would require URL-pattern detection. The default `dataset="worldpop_2020"` is explicit, deterministic, and matches the M5 demo target. The newer vintages can land as future branches once their URL conventions are stable. Surfaced as **OQ-37-WORLDPOP-VINTAGE-YEAR**. Alternatives considered: latest-vintage autoflip (rejected — URL convention is non-deterministic across vintages, would be a silent-wrong-answer bug); year-as-parameter (the `worldpop_<YEAR>` suffix IS the vintage parameter; consumers who want a specific vintage pass `dataset="worldpop_2020"` etc.).

- **Decision: vintage exposed via the `dataset` string suffix, NOT a separate `vintage_year` parameter.** Rationale: the existing parameter surface uses `dataset: str` to discriminate WorldPop vs ACS. Cramming a `vintage_year: int = 2020` parameter alongside would split orthogonality (vintage applies to both WorldPop and ACS) and add a parameter to the FunctionTool surface. Embedding vintage in the dataset name (`worldpop_2020`, `acs_2022`, future `worldpop_2024`) keeps the parameter surface flat — the LLM picks one identifier per call. This also matches the job-0033 ACS convention (`acs_2022` already encodes the vintage). Surfaced as **OQ-37-VINTAGE-PARAMETER-EXPOSURE** (informational). Alternatives considered: separate `vintage_year` int parameter (rejected — orthogonality break; sigil-in-name is the established convention).

- **Decision: ISO3 routing via heuristic country-envelope table, NOT a real point-in-polygon over Natural Earth admin0.** Rationale: same trade-off as the job-0033 `_state_fips_for_lonlat` heuristic — a real PIP needs a cached shapefile asset + `shapely`/`geopandas` lookup, fine for production but more than this mini job needs (the M5 demo target is Fort Myers, FL — ISO3 `USA`). The 9-country envelope (USA + adjacent Caribbean + CAN/MEX) is enough for the demo + any CONUS-adjacent smoke run; replacement with Natural Earth admin0 is a tracked follow-up alongside the job-0033 TIGER state-PIP follow-up. Surfaced as **OQ-37-ISO3-ROUTING-HEURISTIC**. Alternatives considered: real PIP over Natural Earth (rejected — scope creep; this job's deliberate file-ownership boundary is `fetch_population` only); reverse-geocode via Nominatim (rejected — adds an extra hop and a Nominatim usage-policy charge on every call).

- **Decision: download-then-window instead of caching the entire country file as a long-lived intermediate.** Rationale: an alternative WorldPop strategy is to cache the country GeoTIFF in GCS under `static-365d` keyed by `(iso3, year)` and clip on every call. That would be ~50 MB per (country, year) tuple cached forever-static; a Fort Myers query and a Hurricane Ian query and a CONUS query would all share one country-file cache entry. But it would require a two-stage caching pattern that `read_through` doesn't expose today. The simpler M5-substrate decision is: download the country file to a tmp path inside `_fetch_worldpop_population_bytes`, clip via rasterio, return the windowed COG bytes for `read_through` to cache under the bbox-quantized key. Per-bbox cache entries cost more bandwidth on the cold path but match the existing fetcher pattern. A follow-up can add a two-stage optimization. Surfaced as **OQ-37-COUNTRY-FILE-CACHING-STRATEGY** (informational; non-blocking).

- **Decision: keep the WorldPop COG's `style_preset="continuous_dem"` placeholder, same as ACS.** Rationale: only the seven engine-owned QML presets are in scope (FR-QS-5), and the dedicated population preset is still pending (carried from job-0033's **OQ-33-POPULATION-STYLE-PRESET**). Both branches return the placeholder; the eventual `population_density` or `population_choropleth` preset replaces both in one shot.
