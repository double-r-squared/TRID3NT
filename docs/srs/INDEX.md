# SRS v0.3 — Section Index

This directory is the **canonical editing surface** for the GRACE-2 Software Requirements Specification. Each file below maps to one section or appendix of the SRS; together, concatenated in the order shown, they reproduce `docs/SRS_v0.3.md` byte-for-byte (the monolith is regenerated by `make srs` and preserved only for backward compatibility with the immutable line-references in `reports/complete/`). Sub-agents (specialist runners, reviewers, orchestrator audit subagents) should `Read` the narrow file for the section they need rather than loading the full monolith — load `docs/srs/03-functional-requirements.md` for an `FR-XX-N` citation, `docs/srs/B-assessment-envelope-schema.md` for envelope work, `docs/srs/07-milestones.md` + `docs/srs/02-system-overview.md` for sprint planning, and so on. All amendments land in the narrow files; the user runs `make srs` to regenerate the monolith.

## Loading discipline

**Default:** load the narrow file. Sub-agents should `Read docs/srs/<NN-section>.md` for the section they need, **not** `docs/SRS_v0.3.md`. The narrow files are the canonical editing surface; the monolith is a regenerated convenience.

**When to load a narrow file (the common case):**

- A kickoff cites `FR-WC-4` → load `docs/srs/03-functional-requirements.md` only.
- Schema work on the envelope → load `docs/srs/B-assessment-envelope-schema.md` (plus `02-system-overview.md` if Decision G/N context is needed).
- Orchestrator translating a milestone into a sprint → load `docs/srs/07-milestones.md` + `docs/srs/02-system-overview.md` (Decisions).
- Resolving an OQ → load `docs/srs/06-open-questions.md` + the relevant FR/Appendix file the OQ points at.
- An amendment proposal in a report → cite line-anchored paths in the narrow file (`docs/srs/03-functional-requirements.md` line 412).

**When to load the monolith (`docs/SRS_v0.3.md`) — rare:**

- Reading an immutable historical report under `reports/complete/` that references `docs/SRS_v0.3.md` line N. The monolith is regenerated to those exact line numbers, so historical references stay resolvable.
- Bulk grep / global search for a token across all sections at once (`grep -n FR-TA docs/SRS_v0.3.md`). Note: `grep -rn docs/srs/` works equally well for new searches and is preferred.
- Initial onboarding scan ("what is this project?") — though even here, `docs/srs/INDEX.md` + `01-purpose-and-scope.md` + `02-system-overview.md` is usually sufficient.

**Editing discipline:**

- **All** amendments land in `docs/srs/<NN-section>.md`, never in `docs/SRS_v0.3.md`. After landing, run `make srs` to regenerate the monolith.
- Specialists still **propose** amendments through reports — only the user lands them — but the user lands them in the narrow file now, not the monolith.
- The monolith is kept tracked for backward compatibility with the immutable `reports/complete/` history. CI may verify `make srs && git diff --exit-code docs/SRS_v0.3.md` on PRs to catch drift.

**Lossless guarantee:** the monolith is reproducible byte-for-byte by `cat`-concatenating the parts in INDEX order. No banner is injected into part files (that would break round-trip). The "this file is generated" notice lives in the Makefile recipe output and this INDEX only.
